스테이징된 변경사항을 분석해 `.github/COMMIT_CONVENTION.ko.md` 규칙에 맞는 커밋 메시지를 한국어로 작성하고 커밋합니다.

## 작업 순서

### 0. 컨텍스트 캐시 초기화

이전 대화에서 이미 git 명령을 실행한 경우라도 **반드시 아래 명령을 다시 실행**해서 현재 상태를 새로 읽는다. 이전 툴 결과를 재사용하지 않는다.

### 1. 스테이징 상태 확인

```bash
git diff --cached --name-only
git status --short
```

스테이징된 파일이 없으면 중단하고 안내한다:

> 스테이징된 파일이 없습니다. `git add`로 파일을 추가한 후 다시 실행해주세요.

### 2. 변경 내용 파악

```bash
git diff --cached --stat
git diff --cached
```

- 위 명령을 **항상 새로 실행**하여 최신 스테이징 상태를 기준으로 분석한다.

- `pnpm-lock.yaml`, `package-lock.json`, `yarn.lock` 등 lock 파일 변경은 분석에서 **제외**한다.
- 스테이징된 파일의 실제 diff를 기준으로 변경 의도를 파악한다.

### 3. 스코프(scope) 결정

```bash
git rev-parse --abbrev-ref HEAD
```

- 브랜치명이 `main`, `master`, `develop` 이 아니면 브랜치명을 scope로 사용한다.
- 위 브랜치이거나 detached HEAD면 변경 영역을 scope로 사용한다 (예: `components`, `config`, `build`).

### 4. 커밋 메시지 작성 규칙

**형식**:

```
<emoji> <type>(<scope>): <한국어 요약>

- 세부 변경사항 1
- 세부 변경사항 2
```

**타입 선택 기준**:
| Type | 상황 |
|------|------|
| `feat` | 새로운 기능/컴포넌트 추가 |
| `fix` | 버그, 잘못된 동작 수정 |
| `refactor` | 동작 변경 없이 코드 구조 개선 |
| `style` | 포맷, 세미콜론 등 코드 스타일만 변경 |
| `docs` | 문서, 주석, README, CLAUDE.md 등 |
| `chore` | 빌드, 의존성, 설정 등 기타 |
| `perf` | 성능 최적화 |
| `test` | 테스트 추가/수정 |
| `ci` | CI/CD 설정 |
| `build` | 빌드 시스템 변경 |

**Gitmoji 선택 기준** (변경 성격에 가장 적절한 것 하나만):

- ✨ 새로운 기능/컴포넌트
- 🐛 버그 수정
- ♻️ 리팩토링
- 🎨 코드 구조/포맷 개선
- 📝 문서 추가/수정
- 🔧 설정 파일 추가/수정
- ⬆️ 의존성 업그레이드
- ➕ 의존성 추가
- 🔥 코드/파일 삭제
- 💄 UI/스타일 변경
- ♿️ 접근성 개선
- 🏷️ 타입 추가/수정
- 🩹 간단한 수정
- ⚡️ 성능 향상
- 💚 CI 빌드 수정
- 🚧 작업 진행 중

**작성 원칙**:

- 요약은 한국어, 명령문, 현재 시제 (예: "추가", "수정", "제거")
- 마침표 제외, 대문자 사용 안 함
- 본문은 `-`로 시작하는 목록 형태

### 5. 메시지 출력 및 확인

작성한 커밋 메시지를 먼저 출력하고 사용자에게 확인을 구한다:

> 위 메시지로 커밋하시겠습니까? (수정이 필요하면 알려주세요)

### 6. 커밋 실행

사용자가 확인하면 아래와 같이 커밋한다:

```bash
git commit -m "$(cat <<'EOF'
<작성한 커밋 메시지>
EOF
)"
```
