# Implement Work

작업 설명(work description)을 읽고, 그 안에 적힌 "이미 된 것처럼 보이는" 변경사항을 실제로는 아직 구현되지 않은 상태에서 진짜로 구현하는 스킬. `implement-commit-message`를 대체한다 — 그 스킬은 커밋 메시지 형태(제목+이모지+불릿)로 입력이 고정돼 있었지만, 이 스킬은 입력 형식을 가리지 않는다.

## 언제 쓰는가

- 사용자가 작업 설명을 붙여넣고 "이대로 구현해줘"라고 할 때
- 그 설명이 다음 중 어느 형태여도 상관없다:
  - **구조화된 설명**: 번호/섹션으로 나뉜 목록 (예: `/describe-changes` 같은 명령이 만들어낼 법한, "1. OO 스킬 삭제 2. OO 스킬 생성 3. 주요 개선사항 ..." 식의 구조)
  - **커밋 메시지 스타일**: 제목 + 이모지 + `-` 불릿 (기존 `implement-commit-message`가 다루던 형태 — 여전히 지원)
  - **자연어 문단**: 격식 없이 줄글로 "~를 하고 ~도 바꿔줘" 식으로 쓴 설명

입력이 git 커밋일 필요도, 스테이징된 상태일 필요도 없다. `git log`를 조회하지 않는다 — 사용자가 준 텍스트 자체가 스펙이다.

## 하지 않는 것

- 이미 실제 diff/코드에 반영되어 있는 내용을 다시 구현하지 않는다 (4단계에서 먼저 확인).
- 설명에 없는 범위까지 넓혀서 리팩토링하지 않는다 (아래 "최소 diff 원칙" 참조).

## 절차

### 1. 입력 확보

- 스킬 인자(`args`)로 설명이 넘어왔으면 그것을 스펙으로 쓴다.
- 인자가 없으면 직전 사용자 메시지 전체를 스펙 후보로 보고, 실제로 "작업 설명"인지(완료된 것처럼 서술된 변경 목록/문단인지) 판단한다. 단순 질문이나 잡담이면 스킬을 적용하지 않는다.
- 구조화된 설명(번호/섹션)과 자연어 문단은 처리 방식이 다르지 않다 — 둘 다 2단계에서 항목 단위로 쪼갠다. 자연어 문단은 문장/절 단위로 나눠서 항목화한다.
- 입력 형식만으로 스펙 여부가 애매하면 사용자에게 확인한다.

### 2. 항목 단위 분류

제목/서두는 전체 주제 참고용으로만 쓰고, 실제 작업 단위는 번호 항목/불릿/문장 하나하나다. 각 항목을 다음 중 하나로 분류한다:

- **의존성 변경**: "OO에 XX 의존성/패키지 추가" 류. 대상 워크스페이스와 패키지명을 추출한다.
- **파일/코드 변경**: 특정 파일·함수·컴포넌트를 바꾸라는 지시. 파일 경로가 명시돼 있지 않으면 2.5단계에서 찾는다.
- **새 명령/스킬/스크립트 추가 또는 삭제**: "OO 스킬 생성", "OO 스킬 삭제" 처럼 스킬/명령/스크립트 자체를 다루는 지시.
- **개선사항/원칙 서술**: "최소 diff 원칙 추가", "기존 스타일 유지 원칙 명시" 처럼 결과물이 지켜야 할 속성을 서술하는 항목. 이런 항목은 별도 파일을 만드는 게 아니라, 다른 항목을 구현할 때 지켜야 할 제약으로 취급한다.
- **설명/배경 서술(비실행 항목)**: 앞선 변경의 이유나 결과를 설명할 뿐 독립된 산출물이 필요 없는 항목.

### 2.5. 프로젝트 패턴 조사 (구현 전 필수)

바로 매핑하지 말고, 이 저장소가 비슷한 것을 이미 어떻게 하고 있는지 먼저 찾아본다:

- 비슷한 컴포넌트/스킬/설정이 이미 있는가? (예: 스킬을 만들라는 지시면 `.claude/skills/`의 기존 스킬 구조·frontmatter·문체를 참고)
- 이 저장소의 네이밍, 파일 배치, export 방식, 클래스네임 패턴(`cn()`, `classNames` prop 등)이 무엇인가?
- 대상 파일이 지정되지 않은 항목이라면, 이 조사 결과로 대상을 추론한다. 그래도 못 찾으면 사용자에게 묻는다 (추측으로 새 파일/새 위치를 만들지 않는다).

이 단계를 건너뛰고 바로 3단계로 가지 않는다 — "이미 된 것처럼" 적힌 설명일수록 실제 코드 관례와 어긋나게 구현하기 쉽다.

### 3. What → How 변환

각 항목은 "무엇이 되어야 하는지(what)"만 말하고 있다. 2.5단계 조사 결과를 바탕으로 "어떻게 구현할지(how)"를 구체화한다:

- "간소화" "개선" "강화" 같은 추상적 동사는 실제 코드 레벨의 구체적 조작(어떤 함수를 어떤 패턴으로 바꾸는지, 어떤 줄을 추가/삭제하는지)으로 변환한다.
- 의존성 변경은 `pnpm add`/`pnpm remove` 같은 실제 패키지 매니저 명령으로 처리한다 (버전을 손으로 짜맞추지 않는다). 설명이 정확한 버전을 명시했다면 그 버전을 존중한다.
- 새 스킬/명령 추가·삭제는 2.5단계에서 파악한 기존 구조(frontmatter, 디렉터리 위치, 문체)를 그대로 따른다.
- 변환 결과가 여러 방식으로 해석될 수 있으면(예: "명령어"가 셸 스크립트인지 스킬인지) 추측하지 말고 5단계에서 사용자에게 확인한다.

### 4. 현재 상태 대비 이미 됐는지 확인

각 항목에 대해 실제 파일/저장소 상태를 확인해서 이미 반영된 부분은 "이미 있음"으로 표시하고 구현 대상에서 제외한다. 전부 이미 되어 있다면 사용자에게 알리고 종료한다.

### 5. 계획 요약 및 승인 요청

원본 항목과 실제 구현 계획을 1:1 또는 N:1로 매핑해서 짧게 보여준다. 애매하게 해석했거나 확인이 필요한 항목은 여기서 짚는다 (필요하면 AskUserQuestion). 사용자 승인 전에는 6단계로 넘어가지 않는다.

### 6. 구현

두 가지 원칙을 항상 지킨다:

- **기존 스타일 유지 원칙**: 들여쓰기, 따옴표, import 순서, 네이밍, `cn()`/`classNames` 같은 이 저장소 고유 패턴 등 대상 파일·주변 코드의 기존 스타일을 그대로 따른다. 새 스타일을 도입하지 않는다.
- **최소 diff 원칙**: 설명이 요구한 것만 정확히 구현한다. 지나가다 눈에 띈 다른 부분을 같이 고치거나, 요청보다 넓은 범위로 리팩토링하지 않는다. 같은 목적을 더 적은 변경으로 달성할 수 있으면 그쪽을 택한다.

작업 방식:

- 파일 변경은 Read 후 Edit.
- 새 스킬/명령 추가·삭제는 저장소 기존 구조를 그대로 따른다.
- 셸 명령은 순서대로 실행하되 파괴적이거나 장시간 걸리는 명령은 실행 전 한 번 더 알린다.

### 7. 검증 및 보고

- 수정된 파일을 다시 확인(Read/Grep)해서 반영됐는지 검증한다. 타입체크/빌드가 있는 프로젝트면 관련 범위만 실행해 확인한다.
- 실행한 명령의 성공 여부를 확인한다.
- 원본 설명의 각 항목이 실제로 어떻게 구현됐는지 file:line 단위로 보고한다. git add/commit은 사용자가 별도로 요청하기 전까지 하지 않는다.

## 엣지 케이스

- **구조화된 설명과 자연어가 섞여 있음** (예: 번호 목록인데 각 항목이 문장 여러 개): 각 번호를 상위 항목으로 보고, 그 안의 문장들을 하위 세부사항으로 묶어서 분류한다.
- **대상 파일/컴포넌트를 설명만으로 특정할 수 없음**: 2.5단계 조사로도 못 찾으면 추측해서 새로 만들지 말고 사용자에게 확인한다.
- **버전이 설명에 고정되어 있는데 실제 사용 가능한 버전과 다름**: 설명의 버전을 그대로 쓰되, 설치가 실패하면 사용자에게 알린다.
- **항목이 서로 모순됨**: 임의로 하나를 택하지 말고 사용자에게 확인한다.
- **"개선사항" 항목이 실제로는 독립된 코드 변경을 요구함** (예: "에러 처리 강화"가 실제로는 새 검증 로직 추가를 뜻하는 경우): 재분류해서 파일/코드 변경 항목으로 다시 다룬다.
